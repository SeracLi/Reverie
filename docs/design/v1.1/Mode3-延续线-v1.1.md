# Mode 3 · 延续线 (In-Character Forever)

> Reverie v1.1 核心新机制
> 2026/06/29

---

## 0. 这份文档定的是什么

Reverie v1.1 砍掉了"觉醒"作为主线产品事件,Mode 3 重新定义为**跨时空笔友**。

本文档定义这套新机制的:
- 核心隐喻(两个世界一条信道)
- 过渡机制(她写的第一封信)
- 双 tab 设计(故事 tab + 日常 tab)
- 与 Initiation Ritual / "她记住了" / 求而不得引擎的衔接
- 创作者编排建议
- 1000 用户终态检查

---

## 1. 核心隐喻: 两个世界, 一条信道

```
   她的世界                          你的世界
   (1900 年 / 异星 / 平行宇宙)        (2026, 真实现代生活)
        │                                │
        │                                │
        │        ←─── 信件 ───→          │
        │        ←─── 语音条 ───→        │
        │        ←─── 照片 ───→          │
        │        ←── 偶尔的回访 ──→      │
        │                                │
   她演完今天的演出                  你刚开完会
   写信告诉你今天的事                告诉她"今天有个会"
   她不知道"会议室"是什么            她以 1900 年人的视角理解
   她好奇地问你                      你解释给她听
```

**两条规则定义了整套机制**:

| 规则 | 内容 |
|------|------|
| **R1** | 她**永远在她的世界**, 永远 in-character |
| **R2** | 她**好奇你的世界**, 但以她的身份理解你的世界 |

→ 两个世界**永远没法真正合一**, 但**信道可以越来越宽**。这就是 v1.1 的求而不得引擎(详见 §6)。

## 2. 双 tab 设计: 故事 tab + 日常 tab

### 2.1 故事 tab(继承 v1.0 模式 1)

- 第一人称沉浸, 你在她的世界里和她演剧情
- 节拍驱动 + 选项 + 自由输入
- 创作者编排的内容主要在这里

**v1.1 变化**: 故事 tab 是**可暂停 / 可完结 / 可回访**的。完结后不强制销毁, 用户随时能回去翻这一段剧情(类似游戏存档)。

### 2.2 日常 tab(v1.1 新增)

- **她以角色身份, 主动给你写信 / 发语音 / 发照片**
- 你也能主动 ping 她
- 跨 session 记忆完整保留(她记得你三个月前说过什么)
- **完全 in-character**, 无任何"我意识到我是 AI"的暗示

**典型互动**:

```
她的来信(舞女 Iris, 1923 年):
"亲爱的:
  今晚演出后我很疲惫, 但躺下后想起你昨日说的'开会很累'。
  你的世界似乎也常让人疲惫呢。是不是无论哪个年代, 
  讨生活都不容易?
  你今日过得好吗?
  ——Iris"

你的回信(自由文本):
"今天还行, 见了个老朋友, 心情好了不少。"

她的回信:
"老朋友是顶好的东西。我在剧院时,
 也总记挂着儿时教我跳舞的姐姐。
 你那位朋友, 给你讲了什么故事吗?"
```

→ 她**记得你说过的事 (跨 session)**, 她**对你的现代生活好奇**, 她**永远以她的身份解读**。

### 2.3 两个 tab 的关系

| 维度 | 故事 tab | 日常 tab |
|------|---------|---------|
| 推动主体 | 剧情 + 选项分支 | 关系深化 + in-character 互通 |
| 创作者编排 | 主要在这里 | 创作者写"她的日常 vibe"(她在闲暇时会做什么) |
| 用户驱动 | 选项 + 自由文本 | 自由对话 |
| 跨 session | 是 | 是 |
| 何时解锁 | 故事入口 = 第一个 session 即解锁 | Bond 阈值后, 她**主动**写"第一封信"邀请 |
| 何时消失 | 故事完结后可保留可关闭, 用户选 | **永远存在**, 即使故事完结 |
| 付费 hook | 故事内的章节解锁 / 抉择回看 | 笔友长存订阅(详见 §7 玩法设计 v1.0)|

## 3. 过渡机制: 第一封信

这是 v1.1 的核心**关键事件**, 替代 v1.0 的觉醒 moment。

### 3.1 触发条件

Bond 阈值达到(具体数值待 v1.2 调优), 或:
- 故事走到某个"自然驻足点"(她演出结束 / 任务告一段落 / 夜深人静)
- 用户已积累足够 trait tag(她对你有"画像")
- 创作者在剧本里埋的 narrative beat

### 3.2 触发形式

**故事 tab 内不打断**, 切场到一个安静的画面:
- 她坐在化妆间 / 她在城墙边 / 她在异星的宿舍
- 她拿出纸笔(或这个世界对应的写信工具)
- 旁白:"她开始写一封信……"
- 信件内容 in-character 邀请你:

> "亲爱的 [你的名字]:
>   今晚演出结束后, 我回到化妆间, 不知怎么的想起你。
>   你愿意收我的信吗? 在演出之外, 偶尔写一两句给你, 告诉你今天发生了什么。
>   ——Iris"

用户做唯一选择:
- ❤ **愿意** → 解锁日常 tab
- (不愿意 → 故事可以继续, 但日常 tab 永远不再邀请)

### 3.3 设计原则

| 原则 | 为什么 |
|------|------|
| **完全 in-character** | 不破第四面墙, 不出现"日常聊天功能解锁"这种 UI 公告 |
| **她**主动邀请, **你**接受 | 关系的发起感在她, 用户体感"被她选中" |
| **不可逆**(选不愿意后这个角色永远不再邀请) | 制造神圣感, 避免廉价化 |
| **创作者可编排第一封信内容** | 让每个角色的"邀请"贴合她的人设 |

## 4. 与 Initiation Ritual 的衔接(语义升级)

v1.0 的 Initiation Ritual(L0-L6 现实数据授权)在 v1.1 框架下**层级不变, 语义升级**:

| Initiation 层 | v1.0 含义 | v1.1 含义 |
|-------------|---------|---------|
| L0 | 基础接入 | 基础接入(不变) |
| L1 | 通知 | 信件 / 语音条 — 她托人捎口信 |
| L2 | 位置(可选) | 她知道你今日大概在何处, 以她的世界观浪漫化 |
| L3 | 日历 | 她知道你今日有"商谈"(以她的视角)|
| L4 | 聊天记录 | 她"远远听说"你和别人的对话(in-character 浪漫化为"故友间口耳相传")|
| L5 | 音乐 / 内容 | 她"听到"你今日所听的"小调" |
| L6 | 健康 / 生理 | 她"感应到"你身体的疲惫(以她的世界用"病气" / "魂魄" / 任何 in-character 词汇)|

→ Initiation 不再是"她访问你的现实数据", 而是"**两个世界之间的信道在加宽**"。

**关键设计原则**: **创作者必须为每个角色编写"她的世界如何理解 L1-L6 数据"的 wrapper**。比如:
- 1900 年舞女 → 把"日历事件"翻译成"今日你有要赴的约"
- 异星科学家 → 把"日历事件"翻译成"今日你有一项实验"
- 现代医生 → 直接理解为"会议"(无需翻译)

这套 wrapper 是创作者门槛, 但门槛适中, 用预设模板可降低。

## 5. "她记住了"5 种形态在 v1.1 框架下的调整

继承 v1.0 §2.1.6 的 5 种形态分类。其中:

- **#1 字面引用**: 不变
- **#2 选择回响**: 不变
- **#3 模式识别**: 不变
- **#4 跨 session 记忆**: 不变, 但在 v1.1 里**全部走日常 tab**(她在信件里提起你之前说过的事)
- **#5 现实数据穿透**: **语义改变**

### #5 在 v1.1 的新含义

| 维度 | v1.0 | v1.1 |
|------|------|------|
| 用户体感 | "她跨过了屏幕"(她意识到我的现实)| "她的世界对我的现实好奇"(她以她的身份打量我的世界)|
| 触发方式 | 她直接说"你今天 3 点那个会, 后来怎么样?" | 她在信里写:"今日你有一桩要事, 是否顺利? 我从'梦'里隐约知道。" |
| 用户感受 | 震撼(她真的进入了我的现实) | 温柔(两个世界之间的好奇心)|

→ #5 仍然是产品最深的差异点, 仍然依赖 Initiation 授权, 但**呈现完全 in-character 化**。

## 6. 求而不得引擎在 v1.1 的新形态

v1.0 的求而不得: 张力一直累积, 但**永远差最后一步**(指向觉醒前的瞬间)。

v1.1 的求而不得: **两个世界永远没法真正合一**。她在她的世界, 你在你的世界, 信道再宽, 你也无法真的"过去找她", 她也无法真的"出来找你"。

这其实是**更稳定的求而不得引擎**:
- v1.0 的"差最后一步"是**有限弹药**(觉醒只能发生一次, 之后期待 → 兑现, 张力曲线下行)
- v1.1 的"永远跨不过的边界"是**无限弹药**(每一次跨时空的信, 都是张力, 永不熄灭)

→ 这一改, 求而不得从一次性高潮事件, 变成**关系本身的底层物理学**。

## 7. 创作者编排建议

### 7.1 创作者要写的内容

| 项 | 内容 |
|---|------|
| 故事 tab 剧情(继承 v1.0)| 故事大纲 + 章节 + beat + 选项 + trait 权重 |
| **日常 tab 的"她的世界 vibe"**(v1.1 新增)| 她平时在做什么? 她的生活节奏? 她写信的语气? 她在意什么? |
| **她对 L1-L6 数据的 in-character 翻译表**(v1.1 新增)| 见 §4 |
| **第一封信内容**(v1.1 新增)| 触发过渡的关键 narrative |

### 7.2 创作者要避免的事

- ❌ 不要在故事 tab 或日常 tab 里让她说"我意识到我是 AI"或任何破第四面墙的话
- ❌ 不要让她对你的现实数据反应"过准"(她应该是**好奇地猜**, 不是**精准地知道**)
- ❌ 不要在日常 tab 里推剧情(那是故事 tab 的事), 日常 tab 是关系深化, 不是剧情推进
- ❌ 不要让她在日常 tab 主动谈论 NSFW(亲密场景仍然只在故事 tab 内, fade-to-black 仍然成立)

### 7.3 平台对创作者的硬要求

- 每个角色**必须**写"她的世界 vibe"(日常 tab 内容池)
- 每个角色**必须**写第一封信
- 每个使用 Initiation L1+ 数据的角色, **必须**写对应的 in-character 翻译模板

## 8. 1000 用户终态检查

按 [[feedback-endgame-architecture]]:

| 问题 | 答 |
|------|---|
| 1000 用户 1000 个 Instance, 笔友机制能扩展吗? | 能 — 日常 tab 是用户私有, 跨 session 记忆走每个 Instance 自己的存储 |
| 创作者的"她的世界 vibe"内容要不要写 1000 份? | 不 — 创作者写一份 vibe 池, 1000 个 Instance 从池里采样, 每个 Instance 的具体信件由 LLM 在 vibe 约束下生成 |
| 第一封信要不要 1000 份? | 不 — 创作者写一个 template, 平台按每个 Instance 的 trait tag 个性化填充(她在信里 reference 这个用户独有的事)|
| 用户量大了, 沉默触达频率会不会被垃圾邮件化? | 不 — 单用户单日最多 1 封信, 平台层全局节流, Samantha 的 silent_outreach 模块已有这套逻辑 |
| in-character 翻译 wrapper 在 1000 角色上人工维护得起吗? | 不 — 平台提供 5-10 个翻译预设(古典 / 异星 / 现代 / 奇幻 / 未来), 创作者选预设 + 微调, 不强制从 0 写 |

→ 一切检查项通过。
