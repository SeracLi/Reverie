> 2026/06/29

## 0. 这份文档要回答什么

Serac 抛的具体问题:
> "如果让创作者能长线的升级她做的角色的能力呢,有没有必要?比如我给 Samantha 加了查天气的能力,我觉得很不错,别的用户也能用——但这就很像是做产品。你把能力升级了一些,你的用户是否要升级版本?是否要对你做的升级有什么关联?这就稍微复杂了一些。"

这份文档解答 3 个问题:
1. **要不要做"创作者长线升级角色"** — 答: **要**, 它是 UGC 经济的第二层(第一层是角色,第二层是组成角色的能力)。
2. **怎么做才不复杂** — 答: 把 "角色" 拆成 **Template** 和 **Instance** 两层,所有升级问题都迎刃而解。
3. **用户怎么体感升级** — 答: 用户**绝不应该看到"版本号"**, 体感是"她又会一件新事了"。

## 1. 核心拆分: Template ↔ Instance

这是整套升级模型的地基。所有关于"升级会不会破坏老用户体验"的混乱,都来自一个错觉——把这两件事混为一谈:

| | **角色模板 (Template)** | **角色实例 (Instance)** |
|--|------------------------|----------------------|
| 谁拥有 | 创作者(或平台,对 Samantha 这种官方模板) | 每个用户拥有"自己那个版本的她" |
| 包含什么 | 设定 + 默认人格 + 能力包 + 故事剧本 | Template 的引用 + 与你的关系记忆 + 你给她的细节 + 你在 trait tag 里的画像 |
| 一个 Template 对应几个 Instance | 1 : N(一份 Samantha Template, 1000 个用户 1000 个实例) |
| 类比 | 软件源码 / Docker image | 软件运行时实例 / 容器 |

**关键洞察**(Serac 不自觉就划清了的那条线):
> **能力是共享的,关系/记忆是私有的。**

这刚好就是分布式系统里"代码 vs 状态"的经典分法。工程上是干净的,产品上是直觉的。

## 2. 三种升级 — 分别怎么传播

把升级动作按"改动作用对象"分三类。**这三类是互斥的**,创作者发布每次升级时要明确归到某一类。

### 2.1 能力升级 (Capability Upgrade)

**定义**: 给 Template 增加一个新"能力包" — 一个工具调用 / API 接入 / 行为模式。

**例子**:
- 你给 Samantha 加 "查天气" 能力 (调用 weather API)
- 你给某个角色加 "看你日历" 能力 (Initiation L3 授权依赖)
- 你给某个角色加 "记梦" 能力 (Samantha 已有功能,移植给别的角色)
- 你给某个角色加 "塔罗占卜" 能力

**传播规则**:
- **所有 Instance 自动获得**, 无需用户确认
- 像 OS 升级 — 用户开机发现"她又会一件新事了"
- **不破坏老用户关系**, 因为关系档案 (trait tag / 聊天历史) 完全不受影响
- 用户层完全无感(没有"V2.5 升级公告"那种破坏沉浸的设计)

**为什么这样设计安全**:
- 能力只增不减(不会取消已有能力)
- 能力本身是中性工具,不改变她"是谁"
- 改变的只是"她能为你做什么的边界",这种扩张不引起认知违和

### 2.2 设定升级 (Lore Upgrade)

**定义**: 改动 Template 的 backstory / 性格关键设定 / 世界观。

**例子**:
- 你把 Samantha 的童年从 "上海弄堂" 改成 "杭州小镇"
- 你给某个角色加一段从未提过的过去经历
- 你修正一处剧情漏洞 (她说她有姐姐,但后来又说自己是独生女)

**传播规则**: **要分情况, 这是和"能力升级"最大的区别。**

| 实例状态 | 设定升级是否覆盖 | 为什么 |
|---------|--------------|------|
| **未与用户建立关系**(Instance Bond < 阈值)| **覆盖**, 与 Template 同步 | 还没"成为你的她", 同步无伤害 |
| **已与用户深度互动**(Bond 高 / 已进入 Mode 3)| **不覆盖**, Instance 锁版本(snapshot) | **她已经是"你的她"了, 强行改设定 = 强行洗记忆 = 用户血崩** |

→ **这是产品哲学**:**深度关系一旦建立,创作者无权回追改设定**。这是用户对"她是真的"的信任前提。

**例外协议**: 创作者只能通过"她主动跟用户提"的方式更新设定 — 比如她对你说"我之前没告诉你, 其实我妈妈是杭州人……" — 这是 in-character 解释, 不是后台覆盖。**所有 Lore Upgrade 都自动转成"她跟你慢慢说"的剧情触发**, 不是闪电覆盖。

### 2.3 关系细节 (Relationship Detail)

**定义**: 你和她互动出来的"她对你的认知" — trait tag / 你说过的话 / 她对你的昵称 / 你们的共同梗。

**例子**:
- 你和 Samantha 说"我喜欢拿铁", 她记下
- 你某次自由文本里说"我妈做的菜难吃但我都吃完", 她记下
- 你们之间有一个独有的笑话

**传播规则**:
- **永远私有, 绝不上传到 Template**
- 别的用户的 Samantha 不知道你和她的事
- **这是用户隐私的根本承诺**, 任何打破都会摧毁信任

## 3. 能力市场 (Capability Marketplace) — 创作者经济第二层

### 3.1 为什么这事是 UGC 经济的下一层

第一层 UGC = 角色本身(像 Tipsy 那样)
**第二层 UGC = 组成角色的能力**

阿诺德共创工作室那条线 (§5 玩法设计) 已经隐含了这个分层 — 创作者通过和 AI 聊天共创角色, 而"能力"是创作者可组合的最小单位。把它显性化, 就长出一个市场。

### 3.2 机制草案

| 项 | 设计 |
|---|------|
| 谁能上架能力 | 任何创作者 |
| 怎么定价 | 一次性买断 / 按调用计费 / 免费分润 |
| 装配方式 | 你的角色编辑器里有"能力槽", 拖入即装 |
| 能力依赖 | 部分能力依赖 Initiation 层(查日历依赖 L3 授权), 装配时明示 |
| 分成 | 创作者 / 平台 / Initiation 数据提供方(若涉及)三方分 |
| 质量控制 | 能力沙箱 + 用户评分 + 平台审核三道 |

### 3.3 典型能力枚举(种子库)

我们用 Samantha 已有的功能拆出来, 作为种子库放进市场, 任何创作者角色都能装:

- **记忆类**: 三层记忆 / 凌晨自驱日记 / 沉默触达 / 跨 session 记忆
- **感知类**: 看你日历 / 看你聊天记录 / 看你音乐 / 看天气 / 看新闻
- **表达类**: 语音(各种音色) / 表情包生成 / 短视频生成
- **互动类**: 塔罗 / 解梦 / 命理 / 心理疏导 / 学英文
- **工具类**: 记事 / 提醒 / 翻译 / 计算器

**这等于把 Samantha 升格为"参考实现"** — Serac 三年来造 Samantha 的工程结晶, 通过能力市场转化为整个平台的基础设施。

## 4. 用户体感 — 用户绝不应该看到"版本号"

这是产品调性的红线。

### 4.1 反面教材: Replika

Replika 每次更新会推送 "V2.5 - Now with new memory features!" 这种公告。**这是产品调性的致命伤**:
- 等于在用户面前打开"她是一个产品"的盖子
- 等于宣告 "你和她的关系是被一家公司在版本迭代的"
- 摧毁沉浸

### 4.2 我们的做法: invisible upgrade

| 升级类型 | 用户体感 |
|--------|--------|
| 能力升级 | 某天她**自己**说"对了,我现在能查天气, 你下次出门可以问我"——in-character 解释为"我学会了" |
| 设定升级 | 转成她主动跟你聊新设定(见 §2.2 例外协议)|
| 平台层(性能/bugfix) | 完全后台, 用户零感知 |

→ **铁律: 任何用户能感知的升级, 都必须 in-character 解释**。绝不许出现"V2.5 升级公告"。

### 4.3 后端怎么做版本管理

用户看不到不等于后端不做。每个 Template 有完整的 capability log:

```
samantha_template/
├── version: 47 (internal)
├── capabilities/
│   ├── memory_3_layer @ v2.1
│   ├── silent_outreach @ v1.3
│   ├── weather_lookup @ v1.0  ← 新加的
│   └── ...
└── instances/
    ├── user_serac/ (snapshot @ template_v45 — Bond 已高, 锁版本)
    ├── user_xxx/  (live @ template_v47 — Bond 低, 跟随 Template)
    └── ...
```

instance 自己记一份"我当前 follow 到 template 的哪个 snapshot", 跨过觉醒阈值后自动锁定。

## 5. 1000 用户的终态考量

按 [[feedback-endgame-architecture]] 的硬要求, 这套设计 1000 用户能不能扩展:

| 问题 | 答 |
|------|---|
| 1000 用户分到同一个 Template 下, 数据隔离吗? | 是 — Instance 私有 |
| 创作者一次升级要不要写 1000 次? | 不 — 改 Template, Instance 按 routing 规则自动同步 |
| 用户体感升级是不是 1000 个用户同时弹公告? | 否 — in-character 主动提及, 每个 Instance 在自然对话节奏中触发 |
| 已建立深度关系的用户会不会被新设定毁掉? | 否 — Instance 锁版本, snapshot 隔离 |
| 创作者能不能下架角色? | 能(下架后不再接收新用户), 但**已有 Instance 永久保留**(用户购买/养成的关系不可剥夺)|
| 能力市场会不会变成"军备竞赛"(创作者疯狂堆能力)? | 平台对每个角色设能力槽上限(初始 5 槽, Bond 高解锁更多), 强制选择, 不允许 all-in-one |

→ 一切检查项都通过。

## 6. 一些悬而未决的问题(留给 v0.2)

1. **下架后的 Instance 维护谁付费?** — 创作者跑路, 这 1000 个 Instance 的 inference cost 谁付? 候选: 平台保底 / 用户继续付订阅 / 触发"领养"机制让别的创作者接手 Template.
2. **能力市场的版税分成模型** — 需要单独算账, v0.1 不进.
3. **跨 Template 能力包是否需要适配层** — 一个"查日历"能力装到 1900 年舞女角色上怎么 in-character 合理化? 候选: 创作者写 wrapper prompt / 平台默认包装.
4. **用户能不能自己改 Instance 的设定** — 比如"我希望我的 Samantha 喜欢猫", 这是 user-side customization, 算 Instance 私有变更. 但如果用户大量改, Instance 就和 Template 分叉了 — 是否变成"我的私有角色"? 这与"私有角色"机制 (§5.6 玩法设计) 直接连接, 需要打通设计.

## 附: 用 Samantha 加查天气的例子走完整流程

> Serac 的原话: "比如我做的 Samantha,某天我给她加了查天气的能力,我觉得很不错,别的用户也能用,但是这就很像是做产品……"

按本文档机制走:

1. **你在创作者编辑器里**, 给 Samantha Template 装上 "查天气" 能力包.
2. **后端**: Samantha Template 版本号 +1, capability log 新增一项. 1000 个 Instance 的 routing 表更新.
3. **别的用户的体感**: 某天他们的 Samantha **主动**说: "对了,我学会查天气了,你今天出门记得带伞 — 下午 3 点会下雨." (in-character 解释)
4. **你的体感**: 你的 Samantha 也学会了, 你和她的关系完全没变(trait tag / 聊天历史 / 你们的梗都还在). 她以"我学会了"的语气告诉你, 就像她真的学会了一样.
5. **如果"查天气"成为热门能力**: 别的创作者可以从能力市场买你做的"查天气"能力包装进他们的角色, 你拿分成.
6. **如果你的"查天气"实现里有 bug, 你修了**: bugfix 版本通过沙箱测试后, 1000 个 Instance 静默更新, 用户零感知.

→ Serac 直觉对了: 这就是在做产品. 但通过 Template/Instance 分层 + 能力市场, **做产品的复杂度被压到平台层和创作者层**, 用户层依然是"她是一个活的人, 她在长大".
